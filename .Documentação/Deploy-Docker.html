Deploy em Docker

Subir o Kastor na sua VPS via Portainer (ou docker-compose CLI)

Este documento descreve, do zero, como colocar o Kastor rodando em um servidor Docker existente — convivendo com outros serviços (Chatwoot, e-commerce etc). Cobre construção da imagem, persistência em volume, exposição via Nginx Proxy Manager, migração de dados de uma instalação local, backup, atualização e troubleshooting. Para entender o código por dentro, consulte README.html.

01Visão geral #

O Kastor é um monolito Node.js que serve tanto o backend (Express + SQLite) quanto o frontend estático (SPA vanilla) na mesma porta. Em produção, ele roda como um único container Docker. Não há dependência externa em runtime: nem Postgres, nem Redis, nem worker queue. Tudo o que ele precisa pra persistir vive em um único diretório (/app/data dentro do container), montado em um volume Docker.

┌──────────────────────────────────────────────────────────────┐ │ VPS │ │ │ │ ┌─────────────────┐ ┌──────────────────────────┐ │ │ │ Nginx Proxy │ HTTPS │ Container: kastor │ │ │ │ Manager │ ──────► │ Port 3000 (interno) │ │ │ │ (Cloudflare │ │ Node 22 + SQLite │ │ │ │ ou outro) │ │ │ │ │ └─────────────────┘ │ Mount: /app/data │ │ │ ▲ └──────────┬───────────────┘ │ │ │ │ │ │ │ kastor.seu- ▼ │ │ │ dominio.com ┌──────────┐ │ │ │ │ Volume │ │ │ │ │ kastor_ │ │ │ internet │ data │ │ │ └──────────┘ │ └──────────────────────────────────────────────────────────────┘

Você só interage com 3 coisas:

  • Stack no Portainer — define o container, env vars e o volume.
  • Volume kastor_data — guarda tudo (banco SQLite, uploads, secret de criptografia).
  • Nginx Proxy Manager — recebe o tráfego HTTPS do domínio e encaminha pra porta interna do container.
Não tem build step no frontend. O HTML/CSS/JS é servido como está pelo Express. Você atualiza a app fazendo git pull + recriar o container — nada de webpack/vite.

03Arquivos da stack #

4 arquivos na raiz do repositório fazem toda a mágica acontecer:

ArquivoFunção
DockerfileReceita da imagem: Node 22-alpine, instala deps, expõe porta 3000, define healthcheck.
docker-compose.ymlDefine a stack: 1 service kastor, mapeamento de porta, volume, env vars, healthcheck.
.dockerignoreMantém node_modules/, data/, .env, .git etc fora do build context.
.env.exampleTemplate das env vars. Copie pra .env local OU cole no Portainer.

O que o Dockerfile faz, em prosa

  1. Parte de node:22-alpine (imagem ~80MB). Node 22.5+ é obrigatório porque o backend usa o módulo built-in node:sqlite.
  2. Copia só package*.json primeiro e roda npm install --omit=dev. Isso aproveita o cache do Docker — só re-instala dependências se elas mudaram.
  3. Copia o restante do código (respeitando o .dockerignore).
  4. Cria /app/data e o declara como VOLUME — sinal pro Docker de que ali vão dados que devem sobreviver entre recriações.
  5. Define HEALTHCHECK que faz wget em /. Se voltar 200, container é "healthy".
  6. CMD ["node", "server.js"] sobe a app.

O que o docker-compose.yml define

  • Service kastor com restart: unless-stopped — sobe automático após reboot da VPS.
  • Porta: ${HOST_PORT:-8080}:3000 — externa configurável via env, interna fixa em 3000.
  • Env vars: todas as variáveis do app (lista completa em §04).
  • Volume: kastor_data:/app/data — sobrevive a rebuilds.
  • Healthcheck: cada 30s o Docker verifica se a app responde.

04Variáveis de ambiente #

Defina no Portainer (aba "Environment variables" da Stack) ou via .env no diretório do compose.

VariávelDefaultDescrição
HOST_PORT8080Porta exposta no host. Conflita com Chatwoot se usar 3000.
PORT3000Porta interna do container. Não muda em geral.
KASTOR_DATA_DIR/app/dataOnde fica banco, uploads, secret.bin. Bate com o mount.
PUBLIC_URL—URL pública sem barra final. Usada nos emails de reset.
FLUXO_SECRET(auto)Hex 64 chars pra criptografar sessões. openssl rand -hex 32.
KASTOR_SESSION_DAYS30Dias até a sessão expirar.
SMTP_HOST—Servidor SMTP (Gmail, Sendgrid, etc).
SMTP_PORT587TLS na 587; SSL na 465.
SMTP_USER—Usuário SMTP.
SMTP_PASS—Senha/app password.
SMTP_SECUREfalsetrue só pra porta 465.
SMTP_FROM(deriva)Header From. Ex.: Kastor <noreply@dominio.com>
Mínimo viável: HOST_PORT, PUBLIC_URL, FLUXO_SECRET. Sem SMTP a app sobe normal — só não envia email de reset.
Sobre o FLUXO_SECRET: se você deixar vazio, o app gera um e guarda em data/secret.bin. Funciona, mas se você jogar o volume fora, todas as sessões existentes expiram. Em produção, defina explicitamente e guarde num lugar seguro (cofre de senhas).

05Volume e persistência #

Tudo que o Kastor precisa preservar vive em /app/data dentro do container, montado como volume Docker:

ArquivoConteúdo
data/kastor.dbSQLite — usuários, workspaces, clientes, projetos, fluxos, demandas, schedules, anexos (metadata).
data/kastor.db-wal + -shmArquivos de Write-Ahead Log. Sempre junto com o .db.
data/uploads/Anexos de demandas, avatares, ícones de fluxo (extraídos de base64 pra disco).
data/secret.binChave de criptografia auto-gerada (só se FLUXO_SECRET não foi definido).
data/auth.encTokens de sessão criptografados.

Por default, o compose declara um volume nomeado:

volumes:
  kastor_data:

O nome real do volume na sua VPS vai ser NOMESTACK_kastor_data — onde NOMESTACK é o nome que você deu à Stack no Portainer. Ex.: Stack kastor → volume kastor_kastor_data. Confira com:

docker volume ls | grep kastor
Nunca faça bind mount apontando direto pra uma pasta do host (./data:/app/data) em produção. Permissões podem dar errado entre o usuário do container e o do host. Volume nomeado é o caminho seguro.

06Deploy no Portainer #

Caminho recomendado: cria a Stack a partir do repositório do GitHub. Assim, basta dar "redeploy" pra atualizar.

  1. Tenha o código no GitHub. O repo precisa ter os 4 arquivos da §03 na raiz. Se o repo é privado, gere um Personal Access Token (PAT) no GitHub com escopo repo.
  2. Aponte o DNS do subdomínio (ex.: kastor.seudominio.com) pro IP público da VPS. Tipo A. Pode levar alguns minutos pra propagar — não bloqueia o deploy, só o HTTPS depois.
  3. Portainer → Stacks → Add Stack.
    • Name: kastor (vira prefixo do volume)
    • Build method: Repository
    • Repository URL: cole o URL do seu GitHub
    • Repository reference: refs/heads/main (ou a branch que quiser)
    • Compose path: docker-compose.yml
    • Se o repo é privado: marque "Authentication" e cole o PAT
  4. Environment variables: na seção abaixo, adicione todas as variáveis da §04 que você quer setar. Mínimo: HOST_PORT, PUBLIC_URL, FLUXO_SECRET. Se for usar email, adicione as SMTP_*.
  5. Deploy the stack. O Portainer faz git clone + docker compose up --build -d. Primeira build leva ~1–2min (download da imagem Node + npm install). Próximas reusam o cache e ficam segundos.
  6. Verifique nos logs do container (ícone de página em Containers → kastor → Logs). Você deve ver algo como: fluxo. rodando em → http://localhost:3000. Se ver, tá no ar.
  7. Teste o acesso direto via IP+porta antes de mexer com HTTPS: http://IP_DA_VPS:8080 deve mostrar a tela de login. Credenciais default: admin / admin123 (trocar imediatamente no perfil).
Atualizar a stack depois: Portainer → Stacks → kastor → "Pull and redeploy". Ele puxa as últimas mudanças do GitHub, rebuilda e sobe sem perder o volume. Detalhes em §11.

07Deploy via CLI #

Se preferir terminal puro (sem Portainer):

# 1. Clone o repo na VPS
git clone https://github.com/SEU_USER/fluxo.git kastor
cd kastor

# 2. Crie o .env com base no template
cp .env.example .env
nano .env    # preencha HOST_PORT, PUBLIC_URL, FLUXO_SECRET, SMTP_*

# 3. Suba
docker compose up -d --build

# 4. Acompanhe os logs até ver "rodando em"
docker compose logs -f kastor

Comandos úteis no dia-a-dia:

# Status
docker compose ps

# Parar
docker compose down

# Parar e remover volumes (CUIDADO: apaga os dados)
docker compose down -v

# Rebuild forçado (após mudar Dockerfile)
docker compose up -d --build --force-recreate

# Entrar no container pra debugar
docker compose exec kastor sh
ls /app/data
node --version

08Domínio e HTTPS #

Recomendação: Nginx Proxy Manager (NPM) rodando como container irmão na mesma VPS. Ele cuida do certificado SSL via Let's Encrypt automaticamente.

Configuração no NPM

  1. Hosts → Proxy Hosts → Add Proxy Host.
  2. Domain Names: kastor.seudominio.com (apertar Enter pra confirmar o chip)
  3. Scheme: http · Forward Hostname / IP:
    • Se o NPM e o Kastor estão na mesma rede Docker: kastor (nome do service)
    • Se estão em redes separadas: IP interno da VPS (ex.: 172.17.0.1)
  4. Forward Port: 8080 (= HOST_PORT que você definiu)
  5. Marque Block Common Exploits e Websockets Support (não tem WS hoje, mas se um dia tiver, já tá pronto).
  6. Aba SSL:
    • SSL Certificate: Request a new SSL Certificate
    • Marque Force SSL + HTTP/2 Support + HSTS Enabled
    • Concorde com os termos do Let's Encrypt
  7. Aba Advanced, cole: 
    proxy_set_header X-Forwarded-Proto $scheme;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
client_max_body_size 25M;
    O X-Forwarded-Proto garante que cookies marquem Secure sob HTTPS. O client_max_body_size permite uploads de até 25 MB (default Nginx é 1 MB e quebra envio de anexos).
  8. Save. Em ~30s o NPM emite o cert, e https://kastor.seudominio.com tá no ar.
Cookie sob HTTPS: depois de habilitar HTTPS, atualize o PUBLIC_URL nas env vars pra incluir https:// e restarte o container pra que os links de email saiam corretos.

09Migrar dados de uma instalação existente #

Se você já tem o Kastor rodando local (em ./data/kastor.db) ou em outro servidor, dá pra trazer tudo:

Opção A — Antes do primeiro deploy

  1. Copie a pasta data/ da instalação antiga pra VPS (via scp, rsync, ou upload manual). 
    scp -r ./data user@vps:/tmp/kastor-import
  2. Suba a stack uma vez vazia (vai criar o volume). 
    docker compose up -d
  3. Pare o container (sem remover o volume). 
    docker compose stop
  4. Copie os arquivos pro volume: 
    docker run --rm \
  -v kastor_kastor_data:/dest \
  -v /tmp/kastor-import:/src \
  alpine sh -c "cp -a /src/. /dest/ && chown -R 1000:1000 /dest"
  5. Suba de volta: 
    docker compose start

Opção B — Importar via Portainer (sem CLI)

  1. Suba a stack uma vez (cria o volume vazio).
  2. Portainer → Volumes → kastor_kastor_data → "Browse" — interface web pra subir/baixar arquivos do volume. Suba o kastor.db e a pasta uploads/.
  3. Restart o container.
Versão do schema: a migração automática do código roda no boot. Se o banco antigo é de uma versão muito velha (pre-SQLite), o código tenta migrar db.json pra kastor.db sozinho. Veja os logs de boot pra confirmar.

10Backup e restore #

Backup manual (script)

# Gera um tar.gz com todo o volume — rode no host onde o Docker tá
docker run --rm \
  -v kastor_kastor_data:/data \
  -v "$(pwd)":/backup \
  alpine tar czf /backup/kastor-backup-$(date +%F).tar.gz -C /data .

Vai gerar kastor-backup-2026-06-26.tar.gz no diretório atual. Copie pra fora do servidor (S3, Backblaze, outra VPS).

Restore

# Container PARADO antes de restaurar
docker compose stop kastor

docker run --rm \
  -v kastor_kastor_data:/data \
  -v "$(pwd)":/backup \
  alpine sh -c "cd /data && rm -rf ./* && tar xzf /backup/kastor-backup-XXXX.tar.gz"

docker compose start kastor

Backup automatizado (cron)

Adicione no crontab -e do host:

# Backup diário às 03:00, mantém últimos 7
0 3 * * * cd /opt/backups/kastor && \
  docker run --rm -v kastor_kastor_data:/data -v $(pwd):/backup alpine \
    tar czf /backup/kastor-$(date +\%F).tar.gz -C /data . && \
  ls -t kastor-*.tar.gz | tail -n +8 | xargs rm -f
SQLite com WAL é seguro pra copiar a quente. O comando acima funciona com o container rodando, sem corromper o banco. Mas se for migrar entre máquinas, prefira parar primeiro.

11Atualizar a aplicação #

Via Portainer

  1. Faça git push da branch que a Stack acompanha (geralmente main).
  2. Portainer → Stacks → kastor → Pull and redeploy.
  3. Marque Re-pull image + Prune services.
  4. Confirme. Portainer faz git pull + docker compose up --build -d. ~30s a 1min de downtime durante o restart.

Via CLI

cd /caminho/da/stack
git pull
docker compose up -d --build
O volume sobrevive a updates — banco, uploads e tokens ficam intactos. Migrações de schema rodam automaticamente no boot (ver logs).
Antes de updates grandes (mudança de versão major), faça backup primeiro (§10).

12Logs e healthcheck #

Ver logs

# Últimas 100 linhas, seguindo em tempo real
docker compose logs -f --tail 100 kastor

# Pelo Portainer: Containers → kastor → Logs (botão na barra)

O que esperar nos logs normais

› Importando data/db.json antigo para SQLite… 0 entidades migradas
› Usuário inicial — login: admin | senha: admin123 (altere no Perfil)

  fluxo. rodando em  →  http://localhost:3000

Healthcheck

Docker pinga GET / a cada 30s. Verificar status:

docker inspect --format='{{.State.Health.Status}}' kastor
# Esperado: "healthy"

docker inspect --format='{{json .State.Health}}' kastor | jq

Se ficar unhealthy por 3 checks consecutivos, o Docker marca como tal — útil pro NPM ou pro Watchtower ignorar até recuperar. Container não reinicia sozinho por isso a menos que restart-policy esteja configurada pra on-failure.

13Troubleshooting #

Container sobe mas a página não abre

  • Confira que a porta HOST_PORT não está ocupada: ss -tlnp | grep 8080
  • Acesse direto pelo IP+porta antes de envolver o domínio: http://IP:8080
  • Veja os logs: docker compose logs kastor — erro de Node aparece aqui

"Erro de Node version"

Se ver mensagem sobre node:sqlite não existir, a imagem está usando Node < 22.5. Confira o Dockerfile: a primeira linha deve ser FROM node:22-alpine (ou superior).

Login funciona mas perde sessão a cada restart

O FLUXO_SECRET não está definido e o volume não tá persistindo. Sintomas:

  • Sessões expiram após restart do container
  • Arquivo data/secret.bin some entre boots

Fix: defina FLUXO_SECRET explicitamente nas env vars (openssl rand -hex 32) e confirme que o volume tá montado: docker inspect kastor | grep -A 10 Mounts.

Email de reset não chega

  • Confirme que SMTP_HOST, SMTP_USER e SMTP_PASS estão definidos
  • Gmail/Google Workspace: precisa de App Password, não a senha normal
  • Logs mostram erros do nodemailer com o motivo (auth, conexão, etc)
  • Teste com algo simples: SMTP_HOST=smtp.gmail.com, SMTP_PORT=587, SMTP_SECURE=false

Upload de arquivo grande dá erro 413

O Nginx Proxy Manager limita corpo a 1 MB por default. No Proxy Host → Advanced, adicione:

client_max_body_size 25M;

"Cookie not sent over HTTPS"

Após habilitar HTTPS no NPM, o backend precisa saber que está atrás de um proxy. Já vem configurado, mas confira que o NPM está enviando:

proxy_set_header X-Forwarded-Proto $scheme;

Banco corrompido

SQLite com WAL é robusto, mas em casos raros (kill -9, falta de espaço em disco) pode corromper. Restore o backup mais recente (§10). Pra inspecionar:

docker compose exec kastor sh
cd /app/data
sqlite3 kastor.db 'PRAGMA integrity_check;'

Onde fica o volume no host

docker volume inspect kastor_kastor_data
# Mostra o "Mountpoint", tipo /var/lib/docker/volumes/kastor_kastor_data/_data

Não edite arquivos lá direto enquanto o container roda — use os comandos da §10.

14Checklist de segurança #

  • [ ] Troquei a senha do usuário admin no primeiro acesso
  • [ ] Defini FLUXO_SECRET explicitamente e guardei em cofre de senhas
  • [ ] HTTPS habilitado no NPM com Force SSL + HSTS
  • [ ] PUBLIC_URL aponta pro domínio HTTPS (não pro IP nem HTTP)
  • [ ] Backup automático configurado no cron + cópia off-site
  • [ ] Firewall do host fechando a HOST_PORT pro mundo (só o NPM acessa)
  • [ ] .env da VPS tem permissão 600 e dono root
  • [ ] SMTP usando App Password, não a senha real da conta
  • [ ] Container roda como usuário não-root (Node Alpine já faz isso)
  • [ ] Healthcheck retornando healthy
  • [ ] Webhooks de Discord/Slack só apontam pra canais privados da equipe
Atualização de segurança: docker compose pull + up -d a cada algumas semanas pra trazer patches do Node + Alpine base.
